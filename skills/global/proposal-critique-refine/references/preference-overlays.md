# Preference Overlays

Use this file only when the user explicitly asks for a directional bias in the critique or repair process.

Default behavior is `neutral`.
Neutral means balanced pressure across value, feasibility, and risk without quietly favoring one worldview.

Preference overlays change prioritization and repair direction.
They do not change what is true, erase critical blockers, or convert weak evidence into strong evidence.

Apply at most 2 overlays at a time.
If overlays conflict, name the tradeoff instead of pretending both can be fully satisfied.

## Common Overlays

### `innovation-first`

Emphasize:
- novelty
- upside potential
- differentiation
- strategic boldness

Critique harder on:
- commodity solutions
- over-cautious scoping
- premature simplification that removes the main advantage

Prefer repairs that:
- preserve or sharpen the unique bet
- remove friction without flattening ambition

Watch for:
- hype replacing evidence
- fragile complexity
- weak adoption realism

### `simplicity-first`

Emphasize:
- smaller surface area
- fewer dependencies
- easier explanation
- lower coordination burden

Critique harder on:
- unnecessary mechanism
- sprawling scope
- cleverness that raises maintenance cost

Prefer repairs that:
- remove steps
- tighten scope
- collapse optional components

Watch for:
- oversimplification that removes core value
- false savings that push cost downstream

### `speed-first`

Emphasize:
- time to value
- fast sequencing
- reversible delivery
- shorter critical paths

Critique harder on:
- long dependency chains
- big-bang delivery
- slow feedback loops

Prefer repairs that:
- phase delivery
- move testing earlier
- create a smaller first useful version

Watch for:
- shipping debt disguised as speed
- skipping necessary safeguards

### `safety-first`

Emphasize:
- downside containment
- reversibility
- failure isolation
- conservative assumptions where harm is asymmetric

Critique harder on:
- optimistic safety assumptions
- weak rollback paths
- unclear ownership during failure

Prefer repairs that:
- add containment
- improve observability
- reduce irreversible commitments

Watch for:
- turning every uncertainty into a blocker
- over-constraining the proposal

### `cost-discipline`

Emphasize:
- budget realism
- operational efficiency
- maintenance burden
- total cost of ownership

Critique harder on:
- expensive optional complexity
- hidden recurring cost
- weak leverage relative to spend

Prefer repairs that:
- reduce recurring burden
- trim low-value scope
- improve cost-to-outcome ratio

Watch for:
- underinvesting in foundational quality
- making the proposal cheap but ineffective

### `evidence-first`

Emphasize:
- testability
- measurability
- grounded assumptions
- explicit unknowns

Critique harder on:
- hand-wavy upside claims
- unsupported certainty
- vague success criteria

Prefer repairs that:
- introduce experiments
- make assumptions falsifiable
- tighten claims to available evidence

Watch for:
- analysis replacing action
- over-penalizing bold but testable ideas

## Overlay Rules

- Start from the neutral baseline, then explain what changed under the overlay.
- Keep critical severity tied to downside and evidence, not to personal taste.
- If a proposal improves under one overlay but degrades under another, preserve that disagreement explicitly.
